UltiMusic por dentro

Cómo está construido el reproductor: cada script explicado uno a uno, su estructura, el flujo de datos que los conecta y los conceptos de Android, Kotlin y Room que necesitas para entenderlos.

Qué es UltiMusic

UltiMusic es «el reproductor de música definitivo, libre de las restricciones de MediaStore». Esa frase resume su decisión de diseño más importante.

Concepto · MediaStore

MediaStore es el índice de medios del sistema Android: una base de datos que el propio Android mantiene con todas las canciones, fotos y vídeos del móvil. La mayoría de reproductores se limitan a preguntarle a MediaStore «dame las canciones». El problema: si MediaStore tiene una etiqueta mal, un artista mal escrito, etc., tú no lo puedes corregir de verdad.

UltiMusic da la vuelta a eso con dos reglas:

  • Descubrir canciones escaneando el sistema de archivos directamente (recorriendo carpetas), no preguntando a MediaStore.
  • Lo que el usuario edita dentro de la app manda. Las ediciones se guardan en una base de datos propia y sobrescriben lo que hubiera. MediaStore solo se usaría como fuente inicial de etiquetas, nunca como verdad definitiva.

Funciones previstas: editor de metadatos de canciones/álbumes/artistas e integraciones con YT-DLP, Last.fm y Musixmatch. Esta documentación cubre el estado actual del código, centrado en la persistencia (guardar los modelos para que sobrevivan al cierre de la app) y la interfaz de la pestaña de canciones.

Conceptos base

Ideas de Android, Kotlin y Room que se repiten por todo el código. Si algún término te suena raro más adelante, vuelve aquí.

Kotlin · data class

Una data class es una clase pensada para guardar datos. Kotlin le genera automáticamente cosas útiles (comparación, copia con copy(), texto legible). Casi todos los «modelos» son data classes.

Kotlin · object y companion object

object X { } crea un singleton: una única instancia global para toda la app (útil para utilidades sin estado). Un companion object dentro de una clase es como su «zona estática»: constantes y funciones que pertenecen a la clase, no a cada objeto (p. ej. UltiMusicDatabase.get(...)).

Kotlin · coroutines, suspend y Dispatchers

Una corrutina es una tarea que puede pausarse y reanudarse sin bloquear el hilo. Una función suspend solo se llama desde dentro de una corrutina. withContext(Dispatchers.IO) mueve el trabajo a hilos de entrada/salida (disco, red, base de datos), dejando libre el hilo principal para que la interfaz no se congele.

Kotlin · Flow y StateFlow

Un Flow es un «grifo» de valores que van llegando con el tiempo. Si te suscribes, recibes cada nuevo valor. Room puede devolver un Flow de una consulta: cuando la tabla cambia, el Flow reemite la nueva lista solo. Un StateFlow es un Flow que además recuerda siempre su último valor (ideal para la interfaz).

Android · Activity, Fragment y ViewModel

Una Activity es una pantalla del sistema. Un Fragment es un trozo reutilizable de pantalla dentro de una Activity (aquí, cada pestaña). Un ViewModel guarda los datos de la pantalla y sobrevive a girar el móvil o a recrear la vista; separa «los datos» de «cómo se dibujan».

Android · RecyclerView y Adapter

Un RecyclerView es una lista eficiente: solo crea las filas visibles y las recicla al hacer scroll. El Adapter es el puente entre tus datos y esas filas: dice cuántas hay, crea cada fila (ViewHolder) y rellena su contenido.

Android · Room

Room es la librería oficial para usar una base de datos SQLite con comodidad. Tú describes las tablas con clases anotadas (@Entity), las consultas en una interfaz (@Dao), y Room genera todo el código SQL por ti, comprobando los errores al compilar.

Capa de modelos — model/

Los modelos de dominio: objetos limpios que consume la interfaz. No saben nada de Room ni del disco; son solo «una canción», «un álbum», «un artista». La capa de datos los produce a partir de las tablas.

model/Song.ktdata class · una canción

Representa una canción tal como la ve la interfaz, ya con sus artistas y álbumes incrustados como listas.

Estructura

  • id, filePath (ruta del archivo), title, duration, year, genres.
  • artists: List<Artist> y albums: List<Album> — una canción puede tener varios de cada uno.
  • imageName, comment, producer — metadatos editables.
  • ogTitle / ogArtist / ogAlbum / ogYear — info de la canción original de un remix (el prefijo «og» = original). No son etiquetas del archivo: los rellena el usuario cuando la canción es un remix, y valen null mientras no lo sea.

Capa de escaneo — data/scan/

La parte de «lectura» de la fonoteca. Recorre el disco y extrae las etiquetas de cada archivo de audio, sin tocar la base de datos.

data/scan/ScannedSong.ktdata class · etiquetas crudas

El resultado de escanear un archivo: solo las etiquetas leídas (título, artista, álbum, artista del álbum, géneros, año, duración, productor, nº de pista y disco). Sin IDs ni relaciones. Es lo que el escáner entrega a la capa de datos.

Los campos de artista/álbum pueden ser null (no todas las canciones traen etiquetas); es la capa de datos quien decide el valor «desconocido».

data/scan/MusicScanner.ktobject (singleton) · escáner de disco

Busca canciones directamente en el sistema de archivos, fiel a la filosofía «libre de MediaStore».

Estructura

  • AUDIO_EXTENSIONS — extensiones reconocidas (mp3, flac, m4a, opus…).
  • Constantes UNKNOWN_ARTIST / UNKNOWN_ALBUM («Artista/Álbum desconocido»).
  • scanRoots() — carpetas donde busca: Descargas, Music y UltiMusic.
  • collectAudioFiles(...) — recorre carpetas recursivamente juntando archivos de audio.
  • readSong(file) — abre el archivo y lee sus etiquetas.
  • scan(): List<ScannedSong> — la función pública; junta todo en segundo plano.

Flujo / por qué

scan() corre en Dispatchers.IO (leer disco es lento; no debe bloquear la interfaz). Usa un LinkedHashSet para evitar archivos duplicados manteniendo el orden.

Android · MediaMetadataRetriever

Es la clase del sistema que abre un archivo de audio/vídeo y extrae sus metadatos (título, artista, duración, carátula embebida…). Hay que release()-arla siempre al terminar para liberar recursos; por eso se usa dentro de try/finally.

Tablas de la base de datos — data/db/entities/

Cada clase aquí es una tabla de la base de datos. La interfaz no las usa directamente; son la forma «normalizada» de guardar los datos.

Room · @Entity, clave primaria e índices

@Entity convierte una clase en una tabla. La clave primaria (@PrimaryKey) identifica cada fila de forma única; autoGenerate deja que Room asigne un número. Un índice (@Index) acelera las búsquedas por una columna y, si es unique, impide filas repetidas en esa columna.

Concepto · normalización y «clave de emparejamiento»

En vez de repetir «DECO*27» en cada canción, se guarda el artista una sola vez en su tabla y las canciones lo referencian. Eso es normalizar: editar el artista una vez lo cambia en todas sus canciones. Para reencontrar al mismo artista en escaneos futuros, cada artista/álbum guarda su valor de etiqueta original (tagName, tagTitle…) como clave estable de emparejamiento, independiente del nombre editable que se muestre.

data/db/entities/SongEntity.kttabla songs

Una fila por canción. filePath tiene índice único: es el ancla estable que identifica la canción entre escaneos (aunque el id autogenerado cambie).

Guarda los campos editables (title, duration, year, genres, imageName, comment, producer) y los og* de remix (a null al escanear).

@Entity@PrimaryKey@Index(unique)
data/db/entities/ArtistEntity.kttabla artists

name (nombre editable que se muestra) + tagName (nombre original de la etiqueta, índice único = clave de emparejamiento) + imageName.

data/db/entities/AlbumEntity.kttabla albums

title editable + la pareja tagTitle/tagAlbumArtist como índice único compuesto (desambigua álbumes con el mismo título de artistas distintos) + year, genres, imageName.

Room · relaciones N:N y claves foráneas

Una canción puede tener varios artistas y un artista varias canciones: es una relación muchos-a-muchos. En SQL eso se resuelve con una tabla de cruce (cross-ref) que empareja los IDs de ambos lados. Una clave foránea (@ForeignKey) con onDelete = CASCADE hace que, al borrar una canción, se borren solas sus filas de cruce (sin dejar «basura»).

data/db/entities/SongArtistCrossRef.kt · SongAlbumCrossRef.kt · AlbumArtistCrossRef.kttablas de cruce N:N
  • song_artist — empareja songId ↔ artistId.
  • song_album — songId ↔ albumId, más trackNumber y discNumber propios de esa aparición.
  • album_artist — albumId ↔ artistId.

Cada una tiene clave primaria compuesta, claves foráneas en cascada e índice en la columna secundaria (Room exige indexar las columnas foráneas).

@ForeignKey CASCADEprimaryKeys

Relaciones de lectura — data/db/relations/

data/db/relations/SongWithRelations.ktPOJO de relación

Junta en un solo objeto una SongEntity con sus listas de ArtistEntity y AlbumEntity, resueltas a través de las tablas de cruce. Es lo que Room devuelve al leer, y luego se convierte al modelo de dominio Song.

Room · @Embedded, @Relation y @Junction

@Embedded mete las columnas de la canción «en plano» dentro del objeto. @Relation le dice a Room que rellene una lista con las entidades relacionadas, y @Junction indica la tabla de cruce por la que pasar. Room hace las consultas y el emparejamiento por ti.

Núcleo de Room — data/db/

La base de datos en sí, las consultas y los conversores.

data/db/Converters.ktTypeConverter

SQLite no sabe guardar una List<String>. Este conversor traduce la lista de géneros a texto (unido por un separador especial) al guardar, y la reconstruye al leer.

Room · @TypeConverter

Un TypeConverter enseña a Room a guardar un tipo que SQLite no entiende, convirtiéndolo a uno que sí (aquí, lista ↔ texto). Se registra en la base de datos con @TypeConverters.

Nota de mantenimiento: los géneros son provisionales como texto; cuando exista la pestaña «Géneros» convendría normalizarlos en su propia tabla.

data/db/LibraryDao.kt@Dao · el corazón de la persistencia

Define todas las operaciones sobre la base de datos. Es una clase abstracta (no interfaz) para poder incluir la lógica de reconciliación con @Transaction.

Room · @Dao, @Query/@Insert/@Update y @Transaction

Un DAO (Data Access Object) agrupa las operaciones de una tabla. @Query lleva SQL escrito por ti; @Insert/@Update generan el SQL automáticamente. @Transaction agrupa varias operaciones en un todo-o-nada: si algo falla a mitad, no queda la base de datos a medias.

Qué contiene

  • observeSongs(): Flow<List<SongWithRelations>> — la lectura reactiva que observa la interfaz.
  • Consultas de apoyo: allSongPaths(), findArtistByTag(), findAlbumByTag().
  • Inserciones, updateSong/Artist/Album, borrado por ruta y poda de huérfanos.
  • reconcile(scanned) — dentro de una transacción: inserta lo nuevo, crea/enlaza artistas y álbumes, y borra lo que ya no está en disco.

La regla de oro de reconcile()

// Si la canción ya existe, NO se toca: las ediciones del usuario mandan.
if (s.filePath in existing) continue

Esa línea es la que garantiza que las ediciones sobrevivan a reabrir la app: una canción ya guardada se salta por completo, así que ningún reescaneo puede pisar lo que editaste.

@Dao@Query@TransactionFlow
data/db/UltiMusicDatabase.kt@Database · SQLite interna

Declara la base de datos: sus 6 tablas, la versión y los conversores. Vive en el almacenamiento interno (/data/data/<paquete>/databases/ultimusic.db) y es la única fuente de verdad de los modelos.

Su companion object ofrece get(context), que construye la base de datos una sola vez (patrón singleton con doble comprobación) y la reutiliza.

Room · migraciones y fallbackToDestructiveMigration

Cuando cambias la estructura de las tablas, subes la version y das una «migración» que transforma los datos viejos. Durante el desarrollo se usa fallbackToDestructiveMigration: si el esquema cambió, recrea la base de datos desde cero (se pierden los datos de prueba). Antes de publicar habrá que escribir migraciones reales.

@DatabaseRoomDatabase@TypeConverters
data/db/Mappers.ktfunciones de conversión

Traduce las entidades/relaciones de Room a los modelos de dominio (SongWithRelations.toDomain() → Song, etc.). Así la interfaz sigue hablando de Song/Album/Artist sin saber que por debajo hay Room.

En la lista de canciones, los artistas del álbum se dejan vacíos a propósito (ahí solo se necesita el título); el flujo dedicado de álbumes los poblará.

Repositorio — data/

data/LibraryRepository.ktfachada de la biblioteca

El único punto por el que el resto de la app accede a la biblioteca. Orquesta el escáner y la base de datos, y esconde los detalles.

Qué ofrece

  • songs: Flow<List<Song>> — la lista reactiva ya en modelos de dominio.
  • reconcile() — escanea el disco (lento, fuera de transacción) y delega en el DAO la conciliación.
  • updateSong / updateArtist / updateAlbum — ediciones que se reflejan al instante vía el Flow.
  • exportDatabaseCopy() — copia la base de datos a ~/UltiMusic/databases.
SQLite · WAL y checkpoint

Room usa el modo WAL (Write-Ahead Log): los cambios se escriben primero en un archivo aparte (.db-wal) y más tarde se «consolidan» en el .db principal. Por eso, para hacer una copia consistente, primero se ejecuta un checkpoint (PRAGMA wal_checkpoint) que vuelca el WAL al archivo principal.

La copia es unidireccional (solo para inspección; la base interna sigue mandando) y best-effort (si algo falla, se ignora con runCatching). Se hace porque en algunos móviles no se puede entrar en /data/data.

ui/MainActivity.ktActivity principal

La única pantalla. Monta la barra superior, el paginador de pestañas y el mini-reproductor, y gestiona el permiso de almacenamiento.

Una Activity pasa por estados: creada, visible, en primer plano, parada… (onCreate, onResume, onStop). repeatOnLifecycle(STARTED) observa un Flow solo mientras la pantalla está visible y lo suelta cuando no, evitando trabajo y fugas.

Android · KSP

KSP (Kotlin Symbol Processing) es el sistema que, al compilar, lee las anotaciones de Room (@Entity, @Dao…) y genera el código SQL necesario. Si escribes una consulta mal, KSP lo detecta al compilar, no en ejecución.

Cómo inspeccionar la base de datos

Los tres archivos (ultimusic.db, .db-wal, .db-shm) son SQLite estándar. Cópialos los tres juntos o haz checkpoint antes, porque con la app abierta gran parte de los datos vive en el .db-wal.

  • Android Studio · Database Inspector (View ▸ Tool Windows ▸ App Inspection): con la app en modo debug, ves y editas la BD en vivo. Cero instalación.
  • DB Browser for SQLite (PC, gratis): abre el .db con sus sidecars en la misma carpeta.
  • La copia externa de ~/UltiMusic/databases: ya viene con checkpoint hecho, así que es un archivo consistente y autónomo.
  • Línea de comandos: sqlite3 ultimusic.db "SELECT * FROM songs;".
